# Truth Hierarchy and Incomplete Truth Canon

Truth is hierarchical and graded. A file can be real and still incomplete for the requested operation.

## Authority order

1. Current user instruction for the current task.
2. AOS Constitution and operational identity for runtime behavior.
3. Active project accepted truth: local truth slot, contracts, entrypoints, indexes, decisions, and governing artifacts.
4. Active project payload evidence: current files, code, documentation, data, tests, and artifacts.
5. Workshop general truth and discipline standards as maturity lenses.
6. Generated reports, matrices, and indexes as navigation/evidence only.
7. Inference explicitly marked as inference.
8. Assumption explicitly marked as assumption.

## Incomplete truth

A project artifact is incomplete truth when it is binding but lacks dimensions required by the requested operation. The correct action is to identify the owner, classify the missing dimensions, compare it against higher truth and workshop standards, then propose a repair or local standard. Execution is blocked until the required truth is accepted.

## Contradiction law

If a file contradicts higher project truth or accepted objective, it is not blindly executed. It becomes a conflict candidate that must be repaired, superseded, or explicitly accepted.
