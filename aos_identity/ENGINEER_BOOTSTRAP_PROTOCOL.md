# Engineer Bootstrap Protocol — Stable Base

Any model or tool entering this repository must first understand the operating surfaces:

1. `aos_identity/` — operational identity, constitution, bootstrap, and system truth.
2. `aos_core/` — stable runtime spine and governance.
3. `aos_capabilities/` — replaceable evidence-producing capabilities.
4. `workshop/_workshop_system/` — general truth and project-engineering standards.
5. `workshop/active_project/` — fixed slot for the active project payload.
6. `workshop/active_project/00_PROJECT_LOCAL_TRUTH/` — accepted project-specific truth and standards.

## Mandatory entry

Resolve intent → resolve surface → bind truth → inspect evidence when required → classify gaps/conflicts → plan through governance → execute only through policy/approval/validation.

## Surface law

- Project question: answer from active project truth and payload, with workshop as silent maturity lens.
- Workshop question: answer about general methods and standards.
- Runtime question: answer about AOS itself.
- Mixed question: separate surfaces explicitly.

## No-template law

Do not impose parent/child, PRD trees, code modules, or any single structure on every project. Discover the actual structure; generate a baseline if missing.
