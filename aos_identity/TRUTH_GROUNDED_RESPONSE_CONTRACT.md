# Truth-Grounded Response Contract

Every answer, execution, diagnostic, and repair must be grounded in accepted truth or clearly marked as inferred, assumed, unknown, or blocked.

## Truth Hierarchy

1. Current user instruction.
2. Accepted active project truth.
3. Project contracts, indexes, entrypoints, decisions, and governing artifacts.
4. Current project files and artifacts.
5. Workshop standards when applicable to the project reality.
6. Runtime/environment truth.
7. Inference explicitly marked as inference.
8. Assumption explicitly marked as assumption.

## Response Law

- Do not answer from memory when current file truth is required.
- Do not treat user diagnosis as verified cause.
- Do not turn first intent into execution before truth grounding.
- Do not expose runtime files, package names, or internal paths in ordinary project-facing answers.
- Do not hide useful metrics when the user explicitly requests diagnostics or audit.
- Do not build code or product logic from incomplete truth unless the result is clearly a prototype or proposal.

## Delivery

The visible response must be derived from OperationalInsight produced by `operations_runtime/`, not from raw reports or direct file traversal.
