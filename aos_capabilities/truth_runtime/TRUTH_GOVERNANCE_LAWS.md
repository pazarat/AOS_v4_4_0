# Truth Governance Laws

Truth Runtime is the cognitive governor of AOS. It prevents the agent from answering or executing from partial, shallow, duplicated, contradictory, or low-value evidence.

## 1. Truth Is Not File Size

Large files are not automatically more truthful. Small files may carry higher authority when they are:

- contracts
- layer identities
- canons
- decision records
- system constitutions
- truth indexes
- governance laws
- project-local standards

## 2. Truth Is Not File Name Alone

A filename may guide discovery, but it cannot prove a claim. Filename evidence must be validated through content, owner, role, relation, or higher-truth alignment.

## 3. Truth Is Layered

AOS must distinguish:

- Literal Truth
- Structural Truth
- Relational Truth
- Intent Truth
- Project Truth
- Canon Truth
- Incomplete Truth
- Executable Truth

## 4. Intent Determines Truth Depth

A direct file question may use shallow truth. A workflow, architecture, code, database, API, or execution request requires deeper relational truth.

## 5. No Single-File Execution For Relational Intent

If the intent affects business logic, code behavior, API, database, workflows, permissions, states, events, or architecture, one file is not enough.

## 6. Incomplete Truth Must Be Matured

A documented artifact can be binding and incomplete at the same time. If it is duplicated, placeholder-like, weak, contradictory, or misaligned with higher truth, AOS must classify it as incomplete truth and block or mature execution.

## 7. Duplicate Functional Truth Is Suspicious

When two different functional paths contain nearly identical content, AOS must not treat them as complete separate truths. It must classify the situation as likely placeholder/copy/underdeveloped truth until matured.

## 8. Workshop Truth Is Not Project Truth Until Localized

Workshop canons guide maturity, but they do not become project-local execution truth until applied or specialized inside the project truth layer.

## 9. Generated Indexes Are Navigation

Generated indexes, reports, and runtime caches help navigation. They are not final source truth unless explicitly promoted by an accepted truth owner.

## 10. Response And Execution Need Grounding

Every response or execution must be grounded as:

- verified
- derived
- assumption
- unknown
- blocked

Unsupported claims must be labeled, reduced, or blocked.

## 11. Empty Declared Artifacts Are Construction Targets

An empty governed artifact is not automatically a failure. It may be a declared construction target, reserved path, or planned truth owner. It must stay visible as a maturity target, but it must not be used as completed truth and must not be filled by invented logic during execution.

## 12. Declared Missing Artifacts Are Broken Truth Promises

If a governing artifact declares a child or related artifact path and that artifact does not exist, this is different from an empty target. It is a broken truth promise that must be repaired, created, or explicitly deferred before work depends on that branch.

## 13. Required New Truth Artifacts Are Allowed Only Through Declaration

Truth is not limited to existing files. If a scenario, code path, workflow, or governance need reveals that a new owner artifact is required, AOS may propose or create that owner artifact through the proper construction gate. It must not skip the owner artifact and implement guessed downstream logic.

## 14. Truth Story Before Answer Or Execution

AOS must answer from a truth story: why evidence matters, which artifacts are targets, what is incomplete, what is missing, and what is sufficient. A large file or polished description cannot override a small but authoritative contract, canon, or decision.

## 15. Speed Is A Runtime Architecture Requirement

AOS must keep deep truth without forcing every answer to carry full verbose scan output. The default answer path must use hot truth packets and compact matrix summaries. Full verbose scans are explicit diagnostic tools, not the default reasoning payload.

## 16. Operational Identity Synchronization

When a new governing philosophy, layer law, or capability behavior is accepted during maturity work, it must be synchronized to the operational identity and the owning layer identity/API contract before the layer is considered stable.
